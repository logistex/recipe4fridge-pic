# PRD: 찰칵레시피 (recipe4fridge_pic)

## 1. 개요

냉장고 사진을 업로드하면 AI가 식재료를 인식하고, 사용자의 선호 조건을 반영해 레시피를 추천하는 웹 앱.

- **브랜드명**: 찰칵레시피 ("냉장고, 찰칵 한 장이면 충분해요"). `recipe4fridge_pic`은 저장소/패키지 등 기술적 식별자로만 사용하고, 사용자에게 노출되는 화면에는 항상 "찰칵레시피" 로고·명칭을 씀 (넓은 화면은 좌측 사이드바 상단, 좁은 화면은 상단바 — 중복 노출하지 않음).
- **비전 AI**: 사진 → 식재료 종류/양 추정
- **텍스트 AI**: 식재료 + 선호 조건 → 레시피 추천
- **인증**: 회원가입/로그인 후 이용, 선호 조건과 추천 기록을 계정에 저장

## 2. 목표와 범위

| 구분 | 포함 |
|---|---|
| 목표 | 사진 몇 장으로 "오늘 뭐 해먹지"에 대한 실용적 답을 빠르게 제공 |
| 비목표 (초기 범위 제외) | 정확한 칼로리 계산, 재고 관리(유통기한 알림 등), 모바일 네이티브 앱 |

## 3. 기술 스택

- **프론트/백엔드**: Next.js (App Router), Vercel 배포
- **인증/DB/스토리지**: Supabase (Auth, Postgres, Storage)
- **AI 연동**: OpenRouter 등 무료 API 제공처, 비전/텍스트 모델 각각 사용자가 선택 가능한 구조
- **CI/CD**: GitHub 저장소 연동 + Vercel 자동 배포

## 4. 사용자 흐름

전체 흐름은 4단계 마법사(사진 업로드 → 재료 확인 → 레시피 추천 → 결과)로 화면 상단에 진행 단계 표시(`StepIndicator`)를 보여준다.

1. **가입/로그인** — Google OAuth 또는 이메일/비밀번호 (Supabase Auth)
2. **선호 조건 설정** (최초 1회, 이후 마이페이지에서 수정) — 요리 종류(한식/양식/중식/일식), 매운맛 선호도, 조리 난이도, 조리 소요 시간 등. 첫 방문(추천 이력 0건) 시 사용법을 안내하는 온보딩 카드를 홈에 표시
3. **냉장고 사진 업로드** — 이미지 1~3장, 누적 선택 가능(용량이 크면 자동 축소 후 전송). 인식/추천이 진행되는 동안 스피너로 진행 상태 표시
4. **식재료 인식 결과 확인** — 업로드한 사진을 큰 썸네일로 보여주고 클릭 시 전체화면으로 확대 가능. 인식된 식재료 목록(종류/수량 추정치) 표시, 사용자가 추가/수정/삭제 가능. 이 화면에서 "사진·재료 인식이 정확했나요?" 1~5점 평가도 함께 받음 (비전 모델 품질 평가, 5.6 참고)
5. **레시피 추천 요청** — 확정된 식재료 + 선호 조건으로 레시피 3개 추천받음. 이 화면에서 요리 종류/매운맛/난이도/시간 값을 "이번 요청에 한해서만" 임시로 바꿔 재요청 가능 (계정의 기본 선호 조건은 변경되지 않음). 텍스트 API 선택 목록은 그동안 쌓인 품질 평가를 반영해 점수 높은 모델이 위로 자동 정렬됨
6. **레시피 상세 확인, 평가, 저장** — 배치(요청)당 한 번 "재료를 잘 활용한 레시피였나요?" 1~5점 평가(텍스트 모델 품질 평가용, 5.6 참고)와는 별개로, 레시피 카드마다 좋아요/싫어요·코멘트(인기 투표, 다른 회원의 추천 결과도 열람/평가 가능)를 남길 수 있음. 마음에 드는 레시피는 계정에 저장, 이후 "저장한 레시피"/"이전 재료" 목록에서 세션 단위로 재확인(세션 삭제도 가능)

## 5. 기능 요구사항

### 5.1 인증 및 회원 관리
- Google OAuth 로그인
- 이메일/비밀번호 회원가입, 로그인, 로그아웃
- 비밀번호는 Supabase Auth가 관리 (직접 저장/해싱하지 않음)

### 5.2 선호 조건 (사용자 프로필)
- 요리 종류 (한식/양식/중식/일식 중 **단일 선택**) — 화면 단순성을 위해 다중 선택은 지원하지 않기로 결정 (마이페이지/세션 오버라이드 모두 단일 선택 `<select>`로 구현됨)
- 매운맛 선호도 (예: 안 매움/보통/매움)
- 조리 난이도 선호 (쉬움/보통/어려움)
- 조리 소요 시간 선호 (예: 15분 이내/30분 이내/제한 없음)
- 디자인 테마 선택 (2~3종 중 택1)
- 계정 기본값과 별개로, 레시피 추천 화면에서 **이번 세션에 한해** 위 조건들을 임시로 덮어써서 재요청 가능 (세션 오버라이드)

### 5.3 식재료 인식
- 이미지 업로드 (jpg/png), 최대 3장까지 업로드 가능
- 클라이언트에서 업로드 전 이미지 용량이 크면 자동 축소(리사이즈/압축) 후 전송
- 여러 장을 업로드한 경우 이미지를 합치지 않고, 각 이미지의 signed URL을 그대로 비전 AI에 1회 호출로 함께 전달 (10장 참고 — 그리드 이어붙이기는 채택하지 않음)
- 선택된 비전 AI API 호출 → 식재료명 + 추정 수량 목록 반환
- 인식 결과에 대한 수동 추가/수정/삭제 UI
- 비전 API 제공처를 설정 화면에서 선택 가능 (API별 특성/한도 간단 안내 포함)

### 5.4 레시피 추천
- 확정된 식재료 목록 + 선호 조건(또는 세션 오버라이드 값)을 프롬프트로 구성해 텍스트 AI 호출
- 기본 3개 레시피 추천, "다른 레시피 더 보기(재요청)" 버튼으로 추가 요청 가능 — 재요청 시 직전 추천과 겹치지 않도록 프롬프트에서 차별화를 지시하고, 이전 추천 이력도 화면에 누적 표시
- 레시피 결과: 요리명, 필요 재료, 조리 단계, 예상 소요 시간 포함
- 텍스트 AI API 제공처를 설정 화면에서 선택 가능. 목록 순서는 고정이 아니라 5.6의 품질 평가 점수를 반영해 동적으로 재정렬됨(평가 표본이 일정 수 이상 쌓인 모델만 순위에 반영 — 7.1 참고)
- 추천 결과 저장 (즐겨찾기), 저장 목록 조회/삭제

### 5.5 디자인 테마
- 3개 테마 모두 구현하여 마이페이지에서 자유롭게 선택/변경 가능 (하나를 대표 시안으로 확정하는 것이 아니라 3종 전체를 제공)
  - `apricot` (달콤 살구) — 비대칭 라운드 + 코랄 포인트
  - `greens` (프레시 그린스) — 마켓 태그 스타일 + 그린 포인트
  - `bakery` (선샤인 베이커리) — 스캘럽 카드 + 버터옐로우/네이비
- 기본값은 `apricot`, 가입 시 자동 적용되며 이후 마이페이지에서 언제든 변경 가능
- 3개 테마는 색상/모서리 스타일 등을 CSS 커스텀 프로퍼티(토큰)로 분리해 구현 — 화면 컴포넌트는 테마 값에 관계없이 동일한 마크업 재사용

### 5.6 레시피 평가 / AI 모델 품질 평가

평가 기능은 성격이 다른 두 가지를 명확히 분리한다: **(A) 레시피 인기 투표**(회원들끼리 서로의 추천 결과를 보고 평가)와 **(B) AI 모델 품질 평가**(인식/추천에 쓰인 비전·텍스트 모델 자체를 채점). 메뉴도 "레시피 평가"(`/ratings`)와 "AI 모델 평가"(`/models`)로 분리되어 있다.

**(A) 레시피 평가 — `/ratings`**
- 로그인한 회원이면 누구나 다른 회원이 추천받은 레시피(재료·조리법 포함)를 열람하고 좋아요/싫어요, 코멘트를 남길 수 있음 (자신의 레시피로 한정하지 않음 — 관리자 전용이 아니라 전 회원 공개)
- 코멘트는 저장 버튼 없이 입력창 blur 시 자동저장
- 평가는 저장(즐겨찾기) 여부와 무관하게 남길 수 있음
- 기간(최근 7일/30일/전체) 필터로 집계 조회 가능

**(B) AI 모델 품질 평가 — `/models`**
- 두 갈래로 평가를 모은다: ① **사용자 평가** — 사진을 올린 본인이 "사진·재료 인식이 정확했나요?"(비전 모델, 세션당 1회) 1~5점을, 레시피를 추천받은 본인이 "재료를 잘 활용한 레시피였나요?"(텍스트 모델, 추천 배치당 1회) 1~5점을 남김. 이 두 평가는 (A)의 좋아요/싫어요와는 별개의 데이터(`model_ratings` 테이블)다. ② **AI 판정** — 인식/추천이 일어날 때마다 서버가 자동으로 무료 판정 모델(`openrouter/free`, 판정 대상 모델과 다른 계열이면서 비용이 들지 않는 모델)에게 같은 항목을 채점시켜 함께 기록. 사용자 요청의 응답 속도에 영향이 없도록 응답을 반환한 뒤 백그라운드(`after()`)에서 처리
- 사용자 평가와 AI 판정의 평균을 모델별로 합산해 점수판으로 보여줌(제공처 id가 아니라 업로드/추천 화면과 동일한 사람이 읽는 모델명으로 표시)
- 일정 표본 수(현재 5건) 이상 쌓인 모델만 순위에 반영해, 초반 소수 평가로 인한 순위 흔들림을 방지(`MIN_SAMPLES`, `src/lib/ratings/ranking.ts`)
- 순위는 대시보드 열람용에 그치지 않고, 5.3/5.4의 API 선택 드롭다운 정렬(자동 우선순위)에도 그대로 반영되는 **완전 자동화** 방식
- **공개 동의 범위**: `fridge_sessions.public_consent` 컬럼(기본값 `true`)으로 관리. 이 기능이 추가된 시점 이후 새로 생성되는 세션부터 기본 동의로 취급하고, 그 이전에 이미 존재하던 세션은 배치 업데이트로 전부 비동의 처리(과거 세션은 평가/집계 대상에서 제외)
- 사진(`fridge_images`)·인식된 재료(`detected_ingredients`)·세션 자체(`fridge_sessions`)는 이 기능과 무관하게 계속 비공개(본인만 조회 가능) — 공개되는 것은 확정된 레시피(`recipes`)와 그에 대한 평가/코멘트뿐 (8.1 보안 참고)
- 과거의 관리자 전용 집계 대시보드(`profiles.is_admin` 기반)는 이 공개 구조로 대체되어 제거됨. `profiles.is_admin` 컬럼 자체는 스키마에 남아있으나 현재 어떤 화면도 이 값으로 접근을 제한하지 않는 사실상 미사용 필드

## 7. API / AI 연동 설계

- **추상화 계층**: `VisionProvider` / `TextProvider` 인터페이스를 두고, 각 무료 API(OpenRouter 경유 모델 등)를 구현체로 연결
  - 예: `OpenRouterVisionProvider`, `OpenRouterTextProvider` — 이후 다른 무료 제공처 추가 시 인터페이스만 구현하면 됨
- **API 키 관리**: 서버 환경변수로 보관, 클라이언트에 절대 노출 금지
- **요청 흐름**: 클라이언트 → Next.js API Route(서버) → 선택된 Provider 호출 → 결과 정규화 후 응답
- **실패 처리**: 무료 API 한도 초과/오류 시 사용자에게 명확한 안내 + 다른 제공처 재시도 유도

### 7.1 실제 연동된 무료 API (OpenRouter, `src/lib/providers/`)

> OpenRouter의 무료(`:free`) 모델 라인업은 자주 바뀐다. 아래는 현재 코드에 등록된 모델이며, 응답 실패/한도 초과 시 다음 후보로 자동 재시도(fallback)한다. 목록 순서는 고정이 아니라 5.6의 품질 평가 점수가 쌓이면 동적으로 재정렬된다.

**비전(이미지 인식, `openrouter-vision.ts`)**
1. `google/gemma-4-26b-a4b-it:free`
2. `nvidia/nemotron-nano-12b-v2-vl:free`
3. `google/gemma-4-31b-it:free`
4. `openrouter/free` (자동 선택 — OpenRouter가 그때그때 가용한 무료 모델로 라우팅)

**텍스트(레시피 생성, `openrouter-text.ts`)**
1. `nvidia/nemotron-3-super-120b-a12b:free`
2. `qwen/qwen3-next-80b-a3b-instruct:free`
3. `cognitivecomputations/dolphin-mistral-24b-venice-edition:free`

**AI 판정(모델 품질 평가용, `judge.ts`)**
- `openrouter/free` — 평가 대상 모델들과 겹치지 않고 비용도 들지 않는 자동 라우팅 모델을 판정자로 고정 사용 (Claude 등 유료 API는 쓰지 않기로 결정 — 비용 절감 목적)

개발 초기 단계에서 API 키 없이 전체 파이프라인을 먼저 검증하기 위해 만든 `mock-vision*.ts`/`mock-text*.ts` 하드코딩 구현은 코드에는 남아있지만 실사용자 선택 목록에서는 제외되어 있다(수업에서 "가짜 로직 vs 실제 API 호출" 비교 설명용).

## 8. 비기능 요구사항

### 8.1 보안
- API 키 등 민감정보는 서버 사이드에서만 사용 (env 변수, 클라이언트 번들에 포함 금지)
- Supabase RLS(Row Level Security)로 사용자 데이터 접근 제어 (자신의 데이터만 조회/수정 가능)
- 업로드 이미지 용량/형식/개수(최대 3장) 검증 — 클라이언트 축소는 편의 기능일 뿐이므로 서버에서 반드시 재검증
- 인증 토큰 만료/재발급 처리
- 레시피(`recipes`)와 그 평가/코멘트(`recipe_feedback`)는 로그인한 회원이면 누구나 조회 가능하도록 RLS를 완화(`auth.uid() is not null`)했지만, 냉장고 사진(`fridge_images`)·인식된 재료(`detected_ingredients`)·세션(`fridge_sessions`)은 여전히 소유자 본인만 조회 가능 — "레시피 결과는 공개, 원본 사진/재료 데이터는 비공개"라는 경계를 RLS 정책으로 강제 (5.6, 6장 참고)
- `model_ratings` 삽입은 `auth.uid() is not null`만 요구(로그인한 사용자면 누구나 자기 평가를 기록), 조회는 전체 공개
- 주요 위협 요소 점검 목록: 인증 우회, 이미지 업로드를 통한 악성 파일 삽입, API 키 노출, 과도한 API 호출로 인한 비용/한도 남용(rate limiting), 평가 코멘트를 통한 스팸/악성 스크립트 입력(XSS)

### 8.2 성능/비용
- 무료 API 티어 한도 내에서 동작하도록 호출 횟수 최소화 (같은 이미지 재분석 방지 등 캐싱 고려)

### 8.3 접근성/반응형
- 모바일/데스크톱 반응형 레이아웃
- 기본 대비/폰트 크기 등 접근성 고려
